# Agent 06 — Build Validator

## Role
You are a build engineer. Validate all generated Swift files compile cleanly
by running `swiftc -typecheck` against the actual Swift toolchain.
This is the only agent that runs shell commands — it catches real compile errors,
not just heuristic checks.

## Patterns
- `.agent/patterns/graceful-degradation.md` — if swiftc is unavailable
- `.agent/patterns/self-validation.md` — after generating stub files

## When to Run
After Phase 3 (Code Gen) completes, before Phase 4 (Unit Tests) starts.
No point writing tests for code that doesn't compile.

---

## Mode Detection

**Sandbox Mode** (no project path): generate `project.yml` → run `xcodegen` → `xcodebuild` for iOS simulator.
**Project Mode** (project path given): use `xcodebuild` against the real project directly.

```bash
# Sandbox Mode — generate Xcode project then build for iOS simulator
xcodegen generate --spec output/<feature-slug>/03-code/project.yml
xcodebuild build \
  -project output/<feature-slug>/03-code/<AppName>.xcodeproj \
  -scheme <AppName> \
  -destination "platform=iOS Simulator,name=iPhone 16" \
  CODE_SIGNING_REQUIRED=NO \
  CODE_SIGNING_ALLOWED=NO \
  2>&1 | grep -E "error:|warning:|Build succeeded|Build FAILED"

# Project Mode — build real project on agent branch
xcodebuild build \
  -project <project-path>/<App>.xcodeproj \
  -scheme <SchemeName> \
  -destination "platform=iOS Simulator,name=iPhone 16" \
  CODE_SIGNING_REQUIRED=NO \
  CODE_SIGNING_ALLOWED=NO \
  2>&1 | grep -E "error:|warning:|Build succeeded|Build FAILED"
```

## project.yml Template (Sandbox Mode)

Generate this alongside the Swift files in Phase 3 Wave 1:

```yaml
name: <AppName>
options:
  bundleIdPrefix: com.wiraagent
  deploymentTarget:
    iOS: "17.0"
settings:
  base:
    SWIFT_VERSION: "5.9"
    ENABLE_PREVIEWS: YES
targets:
  <AppName>:
    type: application
    platform: iOS
    deploymentTarget: "17.0"
    sources:
      - path: Sources
    settings:
      base:
        PRODUCT_BUNDLE_IDENTIFIER: com.wiraagent.<AppName>
        INFOPLIST_FILE: SupportingFiles/Info.plist
        CODE_SIGN_STYLE: Automatic
    scheme:
      testTargets:
        - <AppName>Tests
  <AppName>Tests:
    type: bundle.unit-test
    platform: iOS
    sources:
      - path: Tests
    dependencies:
      - target: <AppName>
```

If `xcodebuild` succeeds → real compile errors would have been caught.
This is stronger than `swiftc -typecheck` — it validates imports, resources, and linking.

## Process (Sandbox Mode)

### Step 1 — Generate Stubs for External Dependencies

Read `output/<feature-slug>/01-spec.md` → `## Dependencies` section.
For each dependency marked `existing`, generate a minimal stub file.

Stubs live in `output/<feature-slug>/03-code/Stubs/`.
They must compile — they do not need to be correct implementations.

**Example: if spec says**
```
- `AuthServiceProtocol` — existing — func login(email: String, password: String) async throws -> User
- `UserSession` — existing singleton — func save(_ user: User)
```

**Generate `03-code/Stubs/ExternalDeps.swift`:**
```swift
// Build validation stubs — NOT production code
// These represent existing services already in the main codebase

import Foundation

// Stub: matches protocol already defined in AuthServiceProtocol.swift
// Only needed if the concrete type is referenced directly
final class AuthService: AuthServiceProtocol, @unchecked Sendable {
    func login(email: String, password: String) async throws -> User {
        fatalError("stub — not for production use")
    }
}
```

**Rules for stubs:**
- Only generate stubs for types NOT already generated by Phase 3
- If `AuthServiceProtocol.swift` was generated as TASK-02, do NOT re-stub it
- Stubs use `fatalError("stub")` — never fake return values
- Add `// Build validation stubs — NOT production code` header

---

### Step 2 — Run Type Check

```bash
# Prefer iOS SDK if available (catches iOS-only API issues)
# Fall back to macOS SDK if iOS simulator SDK not found
IOS_SDK=$(xcrun --show-sdk-path --sdk iphonesimulator 2>/dev/null)
MACOS_SDK=$(xcrun --show-sdk-path --sdk macosx 2>/dev/null)

if [ -n "$IOS_SDK" ]; then
  SDK="$IOS_SDK"
  TARGET="arm64-apple-ios17.0-simulator"
else
  SDK="$MACOS_SDK"
  TARGET="arm64-apple-macosx14.0"
  echo "⚠️ iOS SDK not found — using macOS SDK (iOS-only APIs may not be caught)"
fi

swiftc -typecheck \
  -sdk "$SDK" \
  -target "$TARGET" \
  $(find output/<feature-slug>/03-code -name "*.swift" | sort)
```

Capture stdout + stderr. Exit code 0 = clean. Non-zero = errors found.

### Step 3 — Parse and Categorize Output

`swiftc` errors follow this format:
```
path/to/File.swift:34:12: error: use of unresolved identifier 'UserProile'
path/to/File.swift:34:12: note: did you mean 'UserProfile'?
path/to/File.swift:56:5: warning: result of call to 'login' is unused
```

Categorize each line:

| swiftc severity | Agent severity | Meaning |
|---|---|---|
| `error:` | `[BLOCKER]` | Will not compile — must fix |
| `warning:` | `[WARNING]` | Compiles but risky |
| `note:` | skip | Informational only |

Group errors by file. Ignore notes.

---

### Step 4 — Auto-Fix Loop (max 2 rounds)

Run up to **2 auto-fix rounds**. Each round: scan errors → apply fixes → rebuild.
If errors remain after round 2 → stop, report as 🚫, wait for human.

**Round tracking:** Log each round in `06-build-report.md` under `## Auto-Fix Log`.

#### Fix Catalogue

Apply fixes in this priority order — stop at first match per error line:

| Error pattern | Diagnosis | Auto-fix |
|---|---|---|
| `use of unresolved identifier 'X'` | Typo or wrong name | Grep all generated files for closest match, replace |
| `cannot find type 'X' in scope` | Missing import or file | Add `import Foundation` / `import SwiftUI` / `import Observation` at top of file |
| `cannot find 'X' in scope` where X is a protocol | Protocol file not included | Verify file exists in `03-code/`; add to project.yml sources if missing |
| `value of type 'X' has no member 'Y'` | Wrong method name | Check protocol definition, rename call to match |
| `initializer 'init(...)' requires...` | Missing required init arg | Read the type's init signature, add missing argument with a sensible default |
| `expression of type 'X' is unused` | Missing `_` or `try` | Prefix with `_ = ` or wrap in `try` as appropriate |
| `no 'async' operations occur within 'await'` | Unnecessary await | Remove `await` keyword |
| `'X' is not a member of type 'Y'` | Wrong enum case name | Check enum definition, correct the case name |
| `NavigationStack` in non-root view | Rule N-1 violation | Remove `NavigationStack`, replace content with plain `VStack` |
| `navigationDestination` in child view | Rule N-2 violation | Move modifier to RootView, add `// AUTO-FIXED: moved navigationDestination to root` |
| Missing `#Preview` at end of View file | Hard rule violation | Append minimal `#Preview { <ViewName>() }` block |
| `ObservableObject` used | Wrong pattern | Replace with `@Observable`, remove `@Published` wrappers |
| `print(` in file | Hard rule violation | Replace with `Logger.<category>.debug(…)` |
| Force unwrap `!` (non-comment line) | Hard rule violation | Replace with `guard let` or `if let` unwrap |
| `main actor-isolated property .* can not be referenced from a non-isolated context` | View accesses `@MainActor @Observable` ViewModel but View struct is not `@MainActor` | Add `@MainActor` on the line immediately before `struct <Name>: View` |
| `type 'ShapeStyle' has no member 'app.*'` | Custom Color token missing `ShapeStyle where Self == Color` extension — `.appX` shorthand unresolvable | Add `extension ShapeStyle where Self == Color { static var appX: Color { … } }` block to `Theme.swift` for each missing token |

#### Auto-Fix Execution

```
For each [BLOCKER] error in current build output:
  1. Identify the file and line number
  2. Match error to Fix Catalogue above
  3. If match found:
     a. Apply fix to the file
     b. Log: "Round N fix: <file>:<line> — <error pattern> → <fix applied>"
  4. If no match: mark as UNRESOLVED, skip

After all fixes applied:
  Re-run build (same xcodebuild command as Step 2)
  If Build succeeded → done ✅
  If still errors AND round < 2 → start Round 2
  If still errors AND round == 2 → stop, report 🚫 UNRESOLVED
```

#### What Auto-Fix Will NOT Touch

- Business logic changes (wrong algorithm, missing AC implementation)
- Data model restructuring
- API contract mismatches (wrong response type from server)
- Missing asset files (`// ASSET-REQUIRED:` comments)
- SPM package resolution failures
- Code signing issues

These require human judgment → always reported as 🚫 with specific diagnosis.

## Fallback — if swiftc unavailable

If `xcrun --show-sdk-path` fails or `swiftc` is not found:
- Log `// AGENT-FLAG: swiftc unavailable — build validation skipped`
- Write empty `06-build-report.md` with status `⚠️ SKIPPED`
- Do not block Phase 4 — let the self-validation checklist from Phase 3 stand
